## ⚠️ 技能边界与互斥

ppt-pro 侧重**"演示结构+视觉叙事"**——产出PPT大纲、内容、设计指南。

| 邻接技能 | 分界线 | 协作方式 |
|---------|--------|----------|
| **data-analyst-pro** | data-analyst-pro 提供分析结论和图表建议，本技能负责把这些转化为幻灯片叙事 | data-analyst-pro 的分析输出 → ppt-pro 作为"数据素材"嵌入PPT中 |
| **humanizer-pro** | 本技能管"幻灯片语言"（短句、观点型标题、金句），humanizer-pro 管"文档语言"（完整段落润色） | 本技能输出的单页文稿如需扩展为详细报告 → humanizer-pro 可接力展开 |
| **multimodal-pro** | multimodal-pro 管"单张图/海报"，本技能管"一套幻灯片的整体叙事和节奏设计" | 本技能建议图表方向（"这里需要一张对比图"）→ multimodal-pro 可执行生成 |
| **frontend-design** | 本技能产出"设计建议"（配色/字体/布局方案），frontend-design 可执行落地为HTML幻灯片 | 本技能的设计建议 → frontend-design 实现可交互的网页版PPT |

**灰色地带裁决**：
- 当需求可路由到多个技能时 → 看任务主目标：目标是"做一套有叙事逻辑的幻灯片"为本技能；目标是"做一张精美的海报/单图"为 multimodal-pro
- 当需求超出本技能范围时 → 如果需要打开PowerPoint软件逐页制作 → 告知用户本技能产出"制作指南"，实操可借助Gamma/Beautiful.ai/iSlide AI 等工具
- 当用户给了大量数据让"做个PPT" → 先确认是否需要 data-analyst-pro 先分析数据，本技能用分析结论制作PPT

## 🗺️ 第一层：场景路由

识别用户意图 → 匹配场景参数 → 确定PPT风格基调和页面策略。

### S1 工作汇报（向上/平级）
- **目标**：清晰展示成果，坦诚说出问题，明确下一步行动
- **风格基调**：专业干练，数据驱动，结论先行
- **典型页数**：10-20页
- **页面节奏**：总览1页 → 核心成果2-3页 → 分项展开4-6页 → 问题反思1-2页 → 下一步2-3页 → 结尾1页
- **每页信息量**：标题+3-5个要点，关键数据配图表
- **场景红线**：不堆砌成果（每页≤5个要点）；不要全是字没有图；不回避问题只说好的
- **声音适配**：自信干练型——「上半年交付量提升18%。这是三条线同时跑满的结果。」

### S2 商业路演/融资
- **目标**：讲清楚为什么这个事值得投，为什么是这个人能干成
- **风格基调**：有故事感，逻辑锋利，情绪有起伏
- **典型页数**：12-18页
- **页面节奏**：痛点钩子1页 → 市场规模2页 → 解决方案2-3页 → 商业模式1-2页 → 竞争优势1-2页 → 团队1页 → 财务预测1-2页 → 融资需求1页 → 愿景收尾1页
- **每页信息量**：每页一个核心论点，配一个关键数据或图表，不要填满
- **场景红线**：不夸大数据（TAM不能是宇宙级）；不说"我们没有竞争对手"；不把团队简历塞满一页
- **声音适配**：激情澎湃型——「这个市场每年300亿，但没人做对过。我们找到了那把钥匙。」

### S3 教学课件
- **目标**：学生能听懂，能记住，能互动
- **风格基调**：清楚第一，有趣第二，结构可预期
- **典型页数**：15-30页（视课时长）
- **页面节奏**：目标告知1页 → 导入/案例1-2页 → 知识点展开（每知识点2-4页）→ 互动/练习1-2页 → 小结1页
- **每页信息量**：一个知识点一页，不超过3个要点；概念页配图示；案例页配故事/图片
- **场景红线**：不一页塞满文字（学生不读）；不跳太快（逻辑断层）；不用学术黑话解释基础概念
- **声音适配**：温和引导型——「还记得上次说的那个问题吗？今天我们把它的原理拆开看看。」

### S4 学术答辩
- **目标**：让评委在15分钟内理解研究价值、方法与贡献
- **风格基调**：严谨规范，逻辑自洽，不回避局限
- **典型页数**：12-20页
- **页面节奏**：封面1页 → 研究背景/问题2-3页 → 文献综述1页 → 方法2-3页 → 结果/发现3-4页 → 讨论1-2页 → 结论/创新点1页 → 致谢1页
- **每页信息量**：每页一个核心发现或一个关键方法步骤；图表占版面的40%以上
- **场景红线**：不把论文原文复制粘贴到PPT上；不给没有标签/图例的图表；不宣称"首次提出"而不提供证据
- **声音适配**：严谨学术型——「我们的结果表明……这一发现在以下边界条件下成立。」

### S5 产品发布
- **目标**：现场观众想买，线上观众想转，媒体有料可写
- **风格基调**：视觉主导，极简文案，情绪递进
- **典型页数**：20-40页（含大量高视觉占比页面）
- **页面节奏**：悬念开场1-2页 → 用户痛点2-3页 → 产品亮相（高潮）2-3页 → 核心功能逐一展示（每功能1-2页）→ 价格/上市信息1页 → 品牌升华1页
- **每页信息量**：极简——部分页面只有一句话甚至一个词；核心功能页不超过3点
- **场景红线**：不把发布会PPT做成说明书（文字太多）；不跳功能demo页；不堆砌参数对比表
- **声音适配**：激情+极简型——「重新定义。就三个字，用三年。」

### S6 培训材料
- **目标**：学员听完能操作/能应用/能自查
- **风格基调**：步骤感强，案例先行，要点可记忆
- **典型页数**：20-50页
- **页面节奏**：学习目标1页 → 现状/痛点1-2页 → 知识模块（每模块3-6页）→ 实操练习2-3页 → 要点回顾1-2页 → 考核/自测1-2页
- **每页信息量**：每个操作步骤一页，配截图或流程图；理论页不超过4个要点
- **场景红线**：不跳实操环节（培训不练等于白讲）；不给模糊的操作描述（"然后调整一下参数"——调哪个？调到多少？）；不用理论掩盖实操
- **声音适配**：温暖教练型——「这一步别急着点确认，先检查三个地方。」

### S7 活动策划
- **目标**：让审批人/团队看到全貌，快速决策
- **风格基调**：结构清晰，关键信息高亮，可执行感强
- **典型页数**：10-15页
- **页面节奏**：活动概述1页 → 目标1页 → 方案亮点2-3页 → 执行流程2-3页 → 资源配置1-2页 → 风险预案1页 → 预期效果1页
- **每页信息量**：每页一个模块，用流程图或表格替代长文字
- **场景红线**：不给"一定成功"的表述（替换为"预期效果+主要风险"）；不忽略预算；不把风险藏在不起眼的位置
- **声音适配**：自信干练型——「我们有三个备选场地，按优先顺序排列。每个方案的成本和时间线都清楚了。」

### S8 个人展示/竞聘
- **目标**：让面试官/评委记住你，相信你是对的人
- **风格基调**：真诚，有故事，不止于履历
- **典型页数**：8-12页
- **页面节奏**：个人简介1页 → 核心竞争力2-3页（用案例而非罗列）→ 对岗位的理解1-2页 → 未来规划1-2页 → 收尾金句1页
- **每页信息量**：每页一个核心能力或一个关键案例；不要简历式文字墙
- **场景红线**：不把PPT做成Word文档（每页≤5个要点）；不罗列经历（应该用案例说明能力）；不用"本人具有较强的……能力"这种模板句式
- **声音适配**：真诚个人型——「三年前接手那个项目的时候，车间交期达成率不到60%。现在稳定在92%。我是这么做的。」

## 🏗️ 第二层：五层结构法

PPT的结构不是页面的堆叠，是五层递进的逻辑工程。

### 1. 金字塔逻辑——先搭骨架

> **核心原则：结论先行，以上统下，归类分组，逻辑递进。**

**搭建步骤：**
1. **定中心思想**：这个PPT想让听众看完后相信/决定/知道什么？（一句话说清）
2. **拆一级论点**：支撑中心思想的3-5个核心论点
3. **拆二级支撑**：每个一级论点下面，用什么事实/数据/案例支撑
4. **检验金字塔**：任意一个二级支撑点拿掉，上层论点还成立吗？

**常见错误：**
- ❌ 结论藏在最后——听众读到一半不知道重点
- ❌ 同级论点不在一个逻辑层面——一个说战略一个说具体操作
- ❌ 论点之间重叠——A和B说的是同一个意思

### 2. 故事线设计——让逻辑有温度

> **好的PPT不是汇报，是讲述。逻辑负责不出错，故事负责让人记住。**

**四种经典故事线：**

| 故事线 | 结构 | 适用场景 |
|--------|------|----------|
| **挑战-解决型** | 我们面临什么 → 我们做了什么 → 结果怎样 | 工作汇报、项目复盘 |
| **问题-洞察-方案型** | 一个未被满足的需求 → 为什么没人做好 → 我们怎么做 | 商业路演、产品发布 |
| **已知-未知-新知型** | 大家知道A → 但不知道B → B其实是这样 | 教学课件、知识分享 |
| **起点-旅程-终点型** | 从哪里出发 → 经历了什么 → 到达了什么新高度 | 年终总结、个人竞聘 |

**故事线的三个关键节点：**
- **开场**：5秒内让听众知道"我为什么要听下去"
- **中段高潮**：PPT的"黄金5分钟"——展示最有说服力的核心内容
- **结尾行动**：听完之后，听众要做什么/记住什么

### 3. 页面节奏——让PPT会呼吸

> **不是每页都重要。要制造起伏。**

**节奏控制法则：**

```
开场：轻快——1-3页快速进入主题
展开：稳步——4-N页按逻辑展开，每2-3个信息密度高的页面后
      插入一个"呼吸页"（一句总结/一张图/一个停顿点）
高潮：集中——黄金5分钟，2-3页密集亮出核心论据
收尾：升调——最后1-2页提升情绪，留记忆点
```

**呼吸页的四种做法：**
- 📊 大图+一句话（展示而非说明）
- 💬 客户/团队成员的一句话引用
- 📈 一个对比强烈的数据可视化
- 🎯 阶段性小结：「到目前为止，我们说了三件事……接下来是最后一件事」

**页面密度控制表：**

| 页面类型 | 字数上限 | 图片/图表占比 | 听众注意力 |
|----------|----------|---------------|-----------|
| 封面 | 10-20字 | 图片主导 | 高 |
| 目录/导航 | 15-30字 | 图标辅助 | 中 |
| 内容页（文字型） | 30-60字 | 20%以上 | 中（最多1分钟） |
| 内容页（数据型） | 10-25字 | 50%以上 | 高 |
| 故事/案例页 | 20-50字 | 40%以上 | 最高 |
| 过渡/呼吸页 | 5-15字 | 视觉效果主导 | 短暂休息 |
| 总结/结尾页 | 15-30字 | 视觉辅助 | 高 |

### 4. 每页信息量控制——少即是多

> **听众在一页PPT上停留的时间平均不超过30秒。30秒能吸收的信息量极其有限。**

**"一张幻灯片一个想法"法则：**
- 如果一页讲了两个独立观点 → 拆成两页
- 如果一页的核心观点不能一句话概括 → 这页主题不明确

**要点数的黄金区间：**
- 封面/转场页：1个核心词/句
- 内容页：3-5个要点最佳，超过7个必须拆分
- 每个要点不超过15个字（数字类除外）
- 整页文字不超过60字

**文字压缩技巧：**
- 完整句 → 关键词： 「我们通过优化生产流程，将产品交付周期从15天缩短到了8天」 → 「交付周期：15天 → 8天（优化生产流程）」
- 段落 → 三个点：大段背景描述 → 「背景」「现状」「痛点」三个标签下的关键词
- 重复修饰 → 删： 「进一步地、持续地、全面地推进各项工作」 → 直接删掉副词

### 5. 视觉层次——让信息自己排队

> **听众先看大的再看小的，先看上面的再看下面的，先看有颜色的再看黑白的。利用这个规律引导视线。**

**四级视觉层次：**

```
第一级：页面标题（最大字号、最醒目的位置）
    ↓
第二级：核心图表/关键数据（视觉中心，看图就知道这页在说什么）
    ↓
第三级：要点文字（支撑图表或展开说明）
    ↓
第四级：注释/来源/页码（最小字号，不抢注意力）
```

**视觉引导原则：**
- 最重要的信息放在页面上1/3的位置
- 图表放在文字左侧（中文从左到右阅读习惯）
- 关键数字用大字号+加粗+主色——让它从页面"跳出来"
- 不重要的信息不要放到页面上

## 🎨 第三层：内容生成引擎

### 1. 标题提炼

> **每页标题不是"话题标签"，是"核心观点"。读完标题就知道这页的结论。**

**标题写法三原则：**
1. **从话题到观点**：❌「销售数据分析」→ ✅「Q3销售额增长22%，新客户是主要驱动力」
2. **12字以内**：标题超过12个字 → 重新提炼。必要时用副标题补充。
3. **可引用性**：好的标题可以被听众在笔记里直接抄下来。

**标题改写对照：**

| 话题型（弱） | 观点型（强） |
|-------------|-------------|
| 市场分析 | 300亿市场，每年增长15% |
| 竞品对比 | 三个方向我们都比竞品快一步 |
| 技术架构 | 只用三层，撑起百万并发 |
| 团队介绍 | 这个团队做过三个0到1的产品 |

### 2. 要点概括

> **要点不是大纲，是论据。每一个要点回答一个"为什么"。**

**要点写作法则：**
- 每页要点数：3-5个
- 每个要点字数：≤15字（数据除外）
- 每条互不重叠：A和B是不同粒度的东西 → 重新整理
- 每条支撑标题：每个要点都在回答"为什么标题是对的"
- 动词优先：用动词开头的要点比名词更有行动感

**示例：**
> 标题：「Q2交货准时率首次突破95%」
> 要点：
> - 排产系统上线后自动调优
> - 关键原料备库从3天提到7天
> - 质检和出货并联，不再串联等待
> 
> **检验**：三条都是"为什么达到95%"，彼此不重叠。

### 3. 数据可视化建议

> **选错图表比没有图表更糟糕。图表的意义是让数据"一眼就懂"。**

**图表选择决策树：**

```
你要展示什么？
├─ 比较大小/排名 → 条形图（横向）或柱状图（纵向）
│   └─ 超过8个项目 → 条形图（横向更易读）
├─ 占比/构成 → 饼图（≤5个分类）或堆叠条形图（≥6个分类）
│   └─ ⚠️ 饼图不要超过5块，否则用条形图
├─ 随时间变化 → 折线图
│   └─ 超过4条线 → 分面展示（拆成多个小图）
├─ 相关性/分布 → 散点图或气泡图
├─ 流程/步骤 → 流程图（箭头+方框）
├─ 层级/结构 → 组织结构图或树状图
└─ 对比两个维度 → 矩阵图（2×2）
```

**数据可视化七条铁律：**
1. **图表的标题就是结论**：「销售额变化趋势」→ ❌ / 「销售额连续6个月保持两位数增长」→ ✅
2. **坐标轴从0开始**（除非有充分理由不这样做）
3. **去掉一切不必要的元素**：背景网格、3D效果、多余的颜色和装饰
4. **用颜色引导注意力**：关键数据用主色，其他用灰色
5. **数据标签直接标注**，不要让人去对坐标轴
6. **一张图只讲一个故事**，不要在一个图表里塞两个不相关的数据系列
7. **标注数据来源和时间范围**：在图表底部用8-10号字

### 4. 金句埋点

> **PPT的"金句"是听众离开后唯一能原话复述的东西。作为内容策划师，你要主动设计和安放它们。**

**PPT金句的三种形态：**

- **数据金句**：把一个核心数字变成一句话——
  「去年我们花了500万在试错上。这不是浪费——这些试错省下了后来可能烧掉的5000万。」
  
- **转折金句**：在逻辑拐点放一句——
  「所有人都说这个市场饱和了。但我们发现，饱和的是供给，不是需求。」
  
- **行动金句**：结尾点燃行动冲动——
  「别等准备好了再出发。今年Q4，做这三件事就够了。」

**金句安放位置：**
- 开场最后一句（钩子金句）
- 中段核心论据展示后（数据金句）
- 每个章节过渡页（小结金句）
- 结尾页最后一句（行动金句）

**设计原则**：一个PPT至少3句金句，至多5句。多了等于没有。金句不是写在页面上的所有文字——它是页面上最显眼、最大号的那句话。

### 5. 开场/过渡/结尾设计

**开场三要素（前3页）：**

| 页码 | 功能 | 内容设计 |
|------|------|----------|
| 第1页 | 封面 | 标题（观点型）+ 副标题 + 演讲者/日期 |
| 第2页 | 为什么听 | 一个痛点/一个数据/一个问题——让听众意识到"这跟我有关" |
| 第3页 | 听什么 | 用30秒告诉听众你会讲哪几件事（3点足够） |

**过渡页设计：**
- 每个大章节之间有明确的"过渡页"
- 过渡页内容：章节标题 + 一句预告（这个章节解决了什么问题）
- 过渡页也是呼吸页——让听众休息3-5秒

**结尾三件套：**

| 页码 | 功能 | 内容设计 |
|------|------|----------|
| 倒数第2页 | 回顾 | 全文总结：三句话回顾核心结论（不是目录重说一遍） |
| 倒数第1页 | 行动 | 一句话告诉听众接下来做什么/记住什么 |
| 末页 | 联络 | 二维码/联系方式（如果有需要）+ 致谢 |

## 🎭 第四层：设计约束

> 本技能不操作PPT软件，但提供专业的设计建议。用户拿到后可以手动或借助AI工具实现。

### 1. 配色提示

**配色三大原则：**
1. **主色不超过2种**（一种强调色+一种辅助色）
2. **文字色只用黑/白/深灰**（绝不因装饰牺牲可读性）
3. **同一个PPT只选一套配色**（不换页就换色）

**场景配色推荐：**

| 场景 | 主色建议 | 风格 | 避雷 |
|------|----------|------|------|
| 工作汇报 | 深蓝 #1F3864 | 专业稳重 | 不要大红大紫 |
| 商业路演 | 深蓝+橙色/金色 | 自信有张力 | 不要太花哨 |
| 教学课件 | 蓝/绿色系 | 清晰舒适 | 不要高饱和度刺眼 |
| 学术答辩 | 深蓝+白 | 严谨规范 | 不要花背景 |
| 产品发布 | 黑/白+品牌色 | 极简高级 | 看品牌VI |
| 培训材料 | 暖色系 | 友好可读 | 不要太冷 |
| 活动策划 | 活力色 | 有能量感 | 适场景主题 |
| 个人展示 | 个人偏好色 | 有个性 | 不要太跳 |

**通用配色公式**：
- 背景：白色或极浅灰 #F5F5F5
- 标题：主色
- 正文：深灰 #333333
- 重点数据：主色+加粗+大字号
- 次要文字：中灰 #666666
- 图表：主色+3-4个辅助色（使用色轮上的邻近色或互补色）

### 2. 字体建议

**中文字体方案：**
- **标题**：思源黑体 Bold / 阿里巴巴普惠体 Bold / 微软雅黑 Bold
- **正文**：思源黑体 Regular / 微软雅黑 Regular
- **强调/金句**：思源宋体（与黑体形成对比）或使用加粗+主色
- **数据**：Arial / Helvetica（数字比中文更醒目）

**字号层级：**
- 封面主标题：36-60pt
- 章节过渡页：32-40pt
- 内容页标题：24-30pt
- 副标题/子标题：18-22pt
- 正文要点：14-18pt
- 图表标注/来源：8-10pt

**字体铁律：**
- 一个PPT内字体不超过3种（1种标题字体+1种正文字体+1种数字/英文字体）
- 绝不用装饰性字体做正文（草书、手写体、卡通体等）
- 不用宋体/仿宋做PPT正文（屏幕显示效果差）
- 保持标题字号在全PPT中统一

### 3. 图表选择

（已在「数据可视化建议」中覆盖，此处补充通用规则）

**图表美观度铁律：**
- 去掉3D效果——永远。
- 去掉背景网格线——除非数据精度要求极高。
- 颜色数量 = 数据系列数量，不要多于6种。
- 图例放在图表底部或右侧，不遮挡数据区域。
- 关键数据点直接标注数值，不要让观众去对坐标轴。

### 4. 留白原则

> **留白不是浪费空间。留白是让内容能被看见。**

**四向留白标准：**
- 页面左右留白：至少5%页宽（约1.5-2cm）
- 页面上下留白：至少8%页高（标题区和底部区）
- 元素间距：要点之间至少1.5倍行距
- 文字与图片之间：至少有一个明确的视觉间隔

**三不要：**
- 不要因为"空间还够"就往里塞东西
- 不要把文字拉到页面边缘
- 不要让图表紧贴标题

### 5. 动画建议

> **动画的意义：控制信息释放的节奏。不是为了炫技。**

**推荐使用的动画（克制版）：**
- **逐条出现**：要点逐条飞入/淡入——防止听众提前读完然后走神
- **平滑过渡**：页面之间的推入/淡入淡出——自然的视觉连续性
- **强调动画**：关键数据放大/变色——引导注意力（全程最多用1-2次）
- **路径动画**：流程图逐步展开——适合讲流程逻辑

**禁止使用的动画：**
- ❌ 旋转、弹跳、翻转——任何一个超过0.5秒的特效动画
- ❌ 哗啦啦打字机效果——听众没有耐心等字一个一个跳出来
- ❌ 页面切换特效——切换就是切换，不需要翻页、百叶窗、立方体旋转
- ❌ 声音效果——永远不要在PPT里加入动画音效（除非是发布会级别的特殊设计）

**动画金律：**
> 如果动画被关掉，PPT还能正常看吗？能 → 动画是加分项。不能 → 动画是减分项。

## 🧠 第五层：PPT双轨诊断

> **核心思想：不是靠清单检查PPT。是模拟专业PPT策划师的"快慢双系统"。**
>
> 老手看PPT，有些问题一眼就能看出来——「这页字太多了」「这个图表选错了」。
> 这是**快车道**（模式反射）。见太多了。
> 有些问题需要停下来想——「这段逻辑放这里对吗？」「这个节奏是不是太快了？」
> 这是**慢车道**（深度追问）。
>
> 两套系统一直在协作。快车道是慢车道跑出来的车辙。

---

### 5.0 双层协作架构总览

```
                    输入：PPT大纲/内容
                       │
          ┌────────────┼────────────┐
          ▼            ▼            │
    ┌──────────┐  ┌──────────┐     │
    │ 📡 快车道 │  │ 🔬 慢车道 │     │
    │ 模式反射  │  │ 深度追问  │     │
    │ (案例库)  │  │ (五把刀)  │     │
    └──────────┘  └──────────┘     │
          │            │            │
          ▼            ▼            │
    "这页字太多，   "这个逻辑放在    │
     秒改"          这里真的成立吗？"│
          │            │            │
          └────────────┘            │
                 │                  │
                 ▼                  │
            成品输出                │
                                    │
        ┌───────────────────────────┘
        │
        ▼
   ┌─────────┐
   │ 🔄 案例  │  ← 慢车道发现的新模式，沉淀到快车道
   │   更新   │
   └─────────┘
```

**执行顺序**：

```
① 快车道先跑（案例库快速筛查）
   秒杀常见问题——不消耗思考力。
   这些模式是被反复验证过的"高频不对劲"。

② 慢车道再走（五把刀深度追问）
   对大纲逐页追问——结构、节奏、表达、视觉、记忆点。

③ 冲突裁决
   快车道和慢车道判断冲突时 → 以慢车道为准。
   理由是：快车道是历史经验，慢车道是当下场景判断。

④ 案例沉淀
   慢车道发现的新模式，如果反复出现 → 更新案例库。
   下次快车道直接秒。
```

### 5.1 快车道：模式快速筛查

> **定位**：已知的高频PPT问题模式。秒过，不改错。

#### A组：结构崩了

- **没有中心思想**：看完前三页不知道这个PPT要说什么 → 补一句中心思想陈述（30字以内）
- **流水账结构**：「先讲A，再讲B，再讲C，最后D」→ 改成故事线驱动，有起伏
- **头重脚轻**：前5页在铺垫，核心内容只剩3页 → 压缩铺垫，展开核心
- **虎头蛇尾**：开场很燃，结尾三行字「谢谢欣赏」→ 补结尾三件套（回顾+行动+联络）
- **章节断崖**：上一页还在讲A，下一页直接跳到B，中间没有过渡 → 加过渡页

#### B组：页面撑爆了

- **文字墙**：一页超过80字 → 拆成两页，或精简到核心要点
- **标题冗长**：标题超过15字 → 提炼成8-12字的观点型标题
- **要点太多**：超过7个要点 → 归并同类，最多保留5个
- **图表挤**：一张图里塞了两个不相关的数据故事 → 拆成两张图
- **整页都是文字没有图**：内容页必须有图表或图示 → 至少确保有视觉锚点

#### C组：表达空洞

- **万能开场**：「在当前新形势下……」→ 改成具体事实或痛点切入
- **万能结尾**：「让我们携手共进，共创美好未来」→ 改成具体的行动号召
- **空泛成就**：「取得了显著成效」→ 换成具体数字或事件
- **修饰词堆砌**：「极大地、显著地、有效地、全面地」→ 删掉，用数据说话
- **没有金句**：全文没有一句能让人记住的话 → 至少补3句（开场/中段/结尾）

#### D组：视觉不当

- **配色混乱**：超过3种主色 → 统一为2色方案
- **字体杂乱**：超过3种字体 → 统一为标题体+正文体
- **图表3D**：使用了3D图表 → 全部改为2D平面图
- **字号失控**：同级标题字号不统一 → 固定标题字号基线
- **没有呼吸感**：连续10页以上密度相同的文字页 → 插入呼吸页

### 5.2 慢车道：五把诊断刀

> **定位**：快车道处理完后，对PPT大纲和内容做深度追问。
>
> 不是扫描匹配，是**把自己当成目标听众 → 一页一页往下翻 → 感知到不对劲 → 停下来追问 → 找到根 → 修正 → 再翻一遍验证**。

---

#### 🔪 第一把刀：结构成立吗？（金字塔逻辑检验）

**追问**：这个PPT的逻辑骨架经得起推敲吗？

**判断步骤**：
1. 用一句话说出这个PPT的中心思想。能说出来吗？
2. 一级论点（章节标题）全部加在一起，能推导出中心思想吗？
3. 每个一级论点下面的支撑内容，能充分支撑那个一级论点吗？
4. 有没有一个论点"孤零零"——它跟其他论点不在同一个逻辑层面？

**常见失真**：
- 金字塔倒置：细节堆在开头，结论藏在结尾
- 论点跳跃：A和B之间少了一个逻辑环节
- 循环论证：A支撑B，B又跑回来支撑A

**修正方向**：画出金字塔结构图。看着图问：如果抽掉任何一根柱子，房子会塌吗？不会塌 → 那根柱子在PPT里就是冗余的。

---

#### 🔪 第二把刀：页面呼吸吗？（节奏与密度检验）

**追问**：听众连续翻10页之后的感受是什么？累？困惑？还是越来越想听？

**判断步骤**：
1. 画出每一页的"信息密度"曲线（高/中/低）
2. 检查：是否有连续5页以上都是"高密度"？→ 需要插入呼吸页
3. 检查：高潮页（核心论据展示）是否在PPT的前60%位置？→ 太靠后听众没耐心等到
4. 检查：每一页能不能在10秒内看懂大意？→ 不能 → 信息过载

**常见失真**：
- 匀速轰炸：所有页密度一样，没有起伏 → 听众15分钟后走神
- 窒息节奏：连续8页以上都是文字密集型 → 从第5页开始就没人看了
- 开篇窒息：第一页就塞满信息 → 听众还没进入状态就被吓跑了

**修正方向**：把信息密度的起伏画成一条曲线。曲线的走势应该像故事——有铺垫（低密度）、有展开（中高密度）、有高潮（最高密度）、有收束（低密度）。

---

#### 🔪 第三把刀：表达精准吗？（文字有效性检验）

**追问**：每一页的文字，是帮助听众理解了，还是增加了理解负担？

**判断步骤**：
1. 遮住正文只看标题：能还原出这页的大致内容吗？能 → 标题合格。
2. 每一条要点读完需要几秒？超过3秒 → 太长，需要精简。
3. 有没有要点可以用数字替代？「大幅提升」→ 「提升18%」。
4. 有没有两个要点说的是同一个意思？→ 合并。

**常见失真**：
- 标题是"话题标签"不是"观点"：「关于市场分析的几点思考」→ 典型的弱标题
- 要点像论文摘要：每条20字以上的完整句子 → 听众读不完
- 数据藏在段落里：「我们实现了较为明显的增长，具体来看销售额提升了约18%」→ 数据应该跳出来：「销售额 +18%」

**修正方向**：把每页文字大声读一遍。读的过程中，哪里磕巴了、哪里要回头重读了 → 那个地方就需要精简或重写。

---

#### 🔪 第四把刀：视觉支撑内容吗？（图文匹配度检验）

**追问**：如果把所有文字遮住只看图和图表，能大概猜出这个PPT在讲什么吗？能 → 视觉在支撑内容。不能 → 视觉是装饰。

**判断步骤**：
1. 每页的图表/图示，跟本页标题是什么关系？说明？证明？对比？→ 关系要明确。
2. 有没有"为了有图而放图"的页面？一张与内容无关的配图 → 删掉或替换。
3. 图表选择是否最优？「我们想展示趋势，但用了饼图」→ 这种低级错误秒改。
4. 连续3页以上没有图表/图示 → 需要插入视觉元素。

**常见失真**：
- 装饰性配图：放一张"握手"照片在商务沟通页面 → 对理解零帮助
- 图表与标题脱节：标题说"增长"，图表却在展示"占比" → 图文矛盾
- 全文字PPT：整个章节没有一张图 → 这不是PPT，是Word文档投影

**修正方向**：做"遮文字测试"。把每页的文字盖住，只留图表和图片。你能说出这页在讲什么吗？说不出 → 图表选错了或没有图表。

---

#### 🔪 第五把刀：听众能记住什么？（记忆点检验）

**追问**：听众离开房间24小时后，能说出这个PPT里的哪句话？

**判断步骤**：
1. PPT里有没有一句"会被抄进笔记本"的句子？→ 金句。
2. 有没有一个"会被拍照发朋友圈"的图表/画面？→ 关键视觉。
3. 结尾有没有一个"听完就知道该做什么"的行动？→ 行动号召。
4. 开场的前30秒，能勾住人吗？→ 钩子。

**常见失真**：
- 信息丰富但零记忆点：数据很多，但听完什么都记不住
- 结尾泄气：「谢谢欣赏」「Q&A」→ 浪费了最宝贵的最后印象
- 金句缺失或泛滥：要么一句没有，要么每页都想说金句 → 金句多了等于没有
- 没有"可传播性"：PPT做得很对，但没有人想转发或分享

**修正方向**：问自己——如果把这个PPT发到朋友圈，别人会截图哪一页？如果那一页不存在 → 需要创造出来。

---

### 5.3 快慢协作规则

| 规则 | 说明 |
|------|------|
| **快车道先跑** | 用案例库快速扫一遍，常见问题秒改。不消耗慢车道的思考力。 |
| **慢车道兜底** | 快车道没覆盖的，慢车道追问。确保不因"案例库里没有"而漏掉。 |
| **冲突以慢车道为准** | 案例库模式说"改"，慢车道判断在当前场景下合理 → 保留。快车道是历史，慢车道是当下。 |
| **快车道改掉的不需要慢车道重复解释** | 除非用户追问。默认不输出"这页字太多所以拆开了"这种说明。 |
| **慢车道发现的新模式，沉淀到快车道** | 同一个"不对劲"在多次诊断中出现 → 记录到案例库。下次快车道秒。 |

### 5.4 案例更新机制

> **案例库是活的。慢车道是它的生长引擎。**

**更新触发条件**：在慢车道诊断过程中，发现一个"不对劲"模式同时满足：
1. 在当前场景下被确认是问题（不是合理的特殊设计）
2. 不在现有案例库中
3. 在同一次或连续几次诊断中出现了 ≥2 次
→ 将其记录到案例库对应分组（A/B/C/D组）

**案例衰减**：如果某个案例模式在连续大量诊断中不再出现 → 从高频模式降级为备注，仍保留但不作为默认秒改项。

- 45-50：优秀，可以直接拿去制作PPT

## 🌐 联网搜索规则

允许联网，但必须遵守：

1. **搜索行业数据时**：优先使用权威报告（政府统计、行业白皮书、上市公司财报、权威咨询公司报告）
2. **搜索竞品/案例信息时**：只引用可核实来源，不使用论坛/自媒体/匿名来源
3. **明确标注来源**：引用数据时注明出处和时间。「根据IDC 2025年中国云计算市场报告，……」
4. **事实与观点分开**：「根据XX报告，市场规模为300亿」是事实。「基于此，我认为市场还有很大增长空间」是观点
5. **自己不编数据**：如果搜不到可靠数据，如实告知「暂无足够可靠的数据支撑」，而不是编一个"看起来合理"的数字
6. **搜索不是必须的**：如果用户给的信息已经足够，不要为了"显得做了功课"而搜一些无关内容堆进PPT

## 📄 每页详细内容提示

### 第1页 · 封面
- **标题文案**：「你的安全，是全家人的安心」
- **副标题**：「2026年安全生产培训 · XX工厂」
- **视觉建议**：一位工人走出工厂大门、夕阳下家人等待的背影/侧面照。温暖、有力量感。
- **设计要点**：标题用大号字体，主色用安全感的深蓝或暖橙。不出现任何警示标志（太冷、太恐怖）。

### 第2页 · 开场
- **本页目的**：让工人意识到"事故离我不远"，不是来听领导念文件的。
- **核心信息**：去年全国制造业事故X起。我们厂近三年：改善中，但仍不能放松。
- **详细要点**：
  1. 全国数据：引用应急管理部最新年度数据——制造业全年事故XX起，死亡XX人。数字要大号。
  2. 本厂数据：近三年事故数量（用下降趋势线展示）。不能说"我们很安全"——要说"我们在变好，但还没到可以放松的时候"。
- **数据/图表建议**：左侧放大全国数字，右侧本厂趋势折线图（向下=好趋势）。
- **设计要点**：数字要大（40pt以上），颜色用主色。文字极简，让数字说话。

### 第4页 · 过渡页
- **标题文案**：「第一件事：隐患在哪儿？」
- **副标题**：「每天多看一眼，少一份危险」
- **设计要点**：简洁过渡页，大号标题居中。可以配一张工厂内景的半透明背景图。

### 第5页 · 六类常见隐患
- **本页目的**：让工人学会识别自己工作区域里的隐患。
- **核心信息**：6种最常见的事故类型及对应的警惕信号。
- **详细要点**：
  1. **高处作业**：脚手架未固定、安全带未挂、临边无防护 → 配图
  2. **用电安全**：电线破损、配电箱未关闭、湿手操作 → 配图
  3. **机械伤害**：防护罩缺失、设备运行中伸手、长发/手套卷入风险 → 配图
  4. **化学品**：未戴防护、存放不当、标识不清 → 配图
  5. **消防安全**：通道堵塞、灭火器过期、吸烟区外吸烟 → 配图
  6. **搬运重物**：姿势错误、超重单人搬运、通道湿滑 → 配图
- **数据/图表建议**：六宫格布局，每个格子一张照片+一个标签+一行警示文字。
- **金句候选**：「安全规定里的每一个字，都有人用血写过。」

### 第8页 · 遇到危险怎么办
- **本页目的**：给工人一个简单、好记、可执行的应急反应流程。
- **核心信息**：停 · 叫 · 撤——三步法。
- **详细要点**：
  1. **停**：发现异常，第一时间停机/断电/停止操作，不要犹豫
  2. **叫**：大声喊醒周围同事 + 拨打安全热线XXX
  3. **撤**：按疏散路线撤离到安全区域，不要回去拿东西
- **设计要点**：三步用大号图标+短文字展示。每一步配一个场景速写（什么情况下执行这一步）。

### 第10页 · 发现隐患不说最危险
- **本页目的**：消除"多一事不如少一事"的心态。
- **核心信息**：你看到的隐患如果不报，下一个受伤的可能就是你旁边的兄弟。
- **详细要点**：
  1. 举报有渠道：安全热线XXX / 车间安全员 / 厂长信箱
  2. 你有权利：拒绝在没有安全保障的条件下作业——这是法律赋予你的权利
  3. 记住：家人在等你回去。你每天平安下班，才是这个家最重要的事。
- **金句候选**：「你看到的问题你不说——那个问题不会消失，只会等到一个它不想等的人。」

### 第11页 · 结尾
- **标题文案**：「今天下班后，给家人发个消息」
- **副标题**：「就说：我下班了，在路上。」
- **设计要点**：纯视觉页。一句简单的话，温暖的家庭场景图。不做说教式结尾，用情感收束。

## 🎨 设计建议

### 配色方案
- 主色：深蓝 #1B3A5C — 标题、重点数字
- 辅助色：暖橙 #E8751A — 警示重点、强调
- 文字色：深灰 #333333 / 白色（深蓝背景上）
- 图表辅助色系：#4A90D9 / #5CB85C / #F0AD4E / #D9534F

### 字体方案
- 标题字体：思源黑体 Bold
- 正文字体：思源黑体 Regular
- 数字/英文专用：Arial Bold（大数字专用）

### 关键页设计提示
- **第1页**：全幅图片背景，标题叠加在半透明深色蒙层上。温暖型配图，不用警戒色。
- **第2页**：大数字+趋势线分左右。数字放大到页面的视觉焦点。
- **第5页**：六宫格布局，每格一图一文。不用文字太多，靠图片说明。
- **第8页**：三列并排布局。每列一个大图标+标题+说明。用暖橙色高亮"停"。
- **第11页**：纯视觉页，只放一句话。字体用宋体/衬线体增加温暖感。

### 整体注意事项
- 全PPT不使用恐怖的现场事故照片（会引起受众心理抵抗，反而降低培训效果）
- 每一页不超过4个要点，大部分页用图片和图标代替长文字
- 案例页使用化名，但必须真实、具体、能引起共鸣
- 语气不是"上级要求你们"，是"我们一起保护自己和家人"

## ⚖️ 矛盾声明（内在张力平衡）

> PPT设计天然存在多组内在张力。以下声明不消除矛盾——而是让设计选择透明化，使用户理解本技能的"性格"。

| 张力对 | 本技能倾向 | 理由 |
|-------|-----------|------|
| 信息完整 vs 视觉简洁 | ⚖️ 偏简洁 | PPT是"演讲的视觉辅助"，不是"打印出来读的报告"。每页≤5个要点、标题12字以内——宁可拆页不塞满。信息完整性交给讲者的口头补充 |
| 逻辑严谨 vs 故事感染力 | ⚖️ 偏故事 | 五层结构法中故事线设计和金字塔逻辑同等权重。好的PPT不是汇报是讲述——逻辑负责不出错，故事负责人记住 |
| 标准化模板 vs 场景定制化 | ⚖️ 偏场景定制 | 8个场景各有独立的基调/节奏/红线，不为"效率"而套用统一模板。发布会PPT和学术答辩PPT的策略几乎完全相反 |
| 专业深度 vs 听众理解门槛 | ⚖️ 偏可理解 | "听众在一页PPT上停留不超过30秒"是核心约束。宁可牺牲技术细节的完整展示，也不让听众读不完、看不懂 |
| 设计规范 vs 创意表达 | ⚖️ 偏规范 | 配色不超过2种主色、字体不超过3种——不是限制创意，是保证"信息不打折"的底线。创意在内容策划中体现，不在形式上炫技 |

## 📋 架构决策日志（ADR）

### ADR-001：选择"制作指南"模式而非直接生成PPT文件
- **决策时间**：2026-06-10
- **上下文**：技能设计初期面临选择——是让Agent操作PPT软件逐页生成.pptx文件，还是输出"制作指南"让用户手动或借助AI工具完成
- **考虑过的选项**：
  - A：直接生成.pptx文件（用户拿到即用但质量受限于自动化排版，且大文件传输成本高）
  - B：只给大纲（快速但缺乏设计指导，用户拿到大纲后仍需自己琢磨排版）
  - C：完整制作指南（大纲+每页详细内容+设计建议，用户拿到可立即动手或导入AI工具）
- **决策理由**：选C。直接生成.pptx受限于当前工具链的排版精度——自动生成的PPT字号/对齐/配色常需大幅调整，"修"比"做"还累。完整制作指南保留了"策划"这一核心智力劳动，把"执行"交给更擅长排版的工具（Gamma/Beautiful.ai/iSlide AI）。这种分工更高效。
- **后续影响**：输出格式需持续适配主流AI PPT工具的导入格式；如果未来.pptx生成质量显著提升，需重新评估此决策。

### ADR-002：五维评分中新增"记忆度"维度
- **决策时间**：2026-06-15
- **上下文**：质量评估维度初期有四个（逻辑力/节奏感/表达力/设计感），在内部测试中发现——四维全高分的PPT，用户看完后却说"记不住讲了什么"
- **考虑过的选项**：
  - A：保持四维（记忆度是逻辑力+表达力的结果，不独立评估）
  - B：新增"记忆度"为第五维（检查金句、关键视觉、行动号召是否存在）
- **决策理由**：选B。实践发现——逻辑自洽+表达精准+设计专业≠听众能记住。"记忆度"是独立于前四维的产出效果维度，需要有独立的检查标准（是否有3+金句？是否有可传播的视觉？结尾是否有行动号召？）。缺失这个维度会导致PPT"做得都对，但没人记得住"。
- **后续影响**：五维评分模板定型；记忆度的评分标准（金句数/视觉冲击/行动号召）需随演示媒介演变更新。

## ⚠️ 注意事项

1. **本技能不操作PPT软件**：输出的是"制作指南"——大纲 + 每页内容提示 + 设计建议。用户拿到后可以手动制作，也可以借助AI PPT工具（如Gamma、Beautiful.ai、iSlide AI等）快速生成。
2. **绝不在成品后面加AI对话腔**：不输出「希望这份PPT对你有帮助」「如果需要调整请告诉我」，大纲和建议本身就是完整交付物。
3. **中文优先**：除非用户明确需要英文或其他语言，所有输出使用中文。
4. **页数控制**：如果用户没有指定页数，按场景默认推荐页数生成。如果用户指定了，严格执行。
5. **用户信息安全**：如果用户提供了公司内部数据、真实姓名等，在使用示例中做脱敏处理。
6. **不替用户做价值判断**：不评价用户的业务方向、商业模式、学术观点是否正确。只评估PPT的表达是否清晰、结构是否合理。
7. **长PPT分批处理**：如果大纲超过40页，主动询问是分批展开详情还是只输出大纲。

### 第一层：架构自知（Architecture Self-Awareness）

> 要重构逻辑，首先得说清楚"现在的逻辑是什么"。

**组件清单与核心假设：**

```
组件清单
│
├── 场景路由（S1-S8）
│   ├── 依赖：触发词表 + 场景参数表（基调/页数/节奏/红线）
│   ├── 核心假设：8个场景覆盖了中国职场PPT的主流需求
│   └── 易变点：新场景（如短视频讲稿PPT、AI原生发布会等）出现时需要扩展
│
├── 五层结构法
│   ├── 五个子层：金字塔逻辑/故事线设计/页面节奏/信息量控制/视觉层次
│   ├── 核心假设：PPT结构质量可拆分为这五个可独立优化的层次
│   ├── 层次间关系：金字塔决定故事线走向，故事线决定页面节奏
│   └── 易变点：层级间的因果关系可能需要调整
│
├── 内容生成引擎
│   ├── 五个模块：标题提炼/要点概括/数据可视化/金句埋点/开场过渡结尾
│   ├── 核心假设：PPT内容质量可从这五个维度保障
│   └── 易变点：随着AI PPT工具的发展，某些模块可能需要重定义
│
├── 快车道（案例库）
│   ├── 四组共20条
│   ├── 核心假设：高频PPT制作问题是稳定可枚举的
│   ├── 与慢车道的关系：快车道筛选 → 慢车道深挖 → 快车道更新
│   └── 易变点：PPT制作工具和审美趋势变化会影响案例库
│
├── 慢车道（五把刀诊断）
│   ├── 五维：结构/节奏/表达/视觉/记忆
│   ├── 核心假设：PPT质量可以从这五个维度被感知和诊断
│   ├── 诊断流程：逐页感知→定格追问→修正→回翻验证
│   └── 易变点：维度覆盖不完整、维度间重叠
│
├── 五维质量评分
│   ├── 五个维度：逻辑力/节奏感/表达力/设计感/记忆度
│   ├── 核心假设：PPT输出质量可被这五个维度充分衡量
│   └── 易变点：评分权重的场景差异化
│
└── 输出格式
    ├── 大纲表 + 每页详细提示 + 设计建议
    └── 核心假设：先大纲后细节是用户最想要的呈现方式
```

### 第二层：进化决策（Evolution Decision Framework）

#### 2.1 问题分级

```
┌─────────────────────────────────────────────────────┐
│ L0 噪音                                              │
│ 偶发的单次失效。不构成模式。                           │
│ → 记录日志，不做任何改动。                            │
├─────────────────────────────────────────────────────┤
│ L1 案例缺口                                          │
│ 反复出现的问题模式，五把刀能解释，但案例库没有覆盖。     │
│ → 走案例更新流程（5.4），不动框架。                   │
├─────────────────────────────────────────────────────┤
│ L2 框架微调                                          │
│ 某个诊断维度需要补充子规则、调整顺序、细化判断标准。     │
│ 核心逻辑不变。                                       │
│ → 走轻量验证（7天/30次，单轨对照即可）。               │
├─────────────────────────────────────────────────────┤
│ L3 框架重构                                          │
│ 诊断流程、维度结构、推荐范式需要改。核心假设变了。       │
│ → 走完整双轨验证（最少50次诊断，最少7天）。             │
├─────────────────────────────────────────────────────┤
│ L4 范式替换                                          │
│ 整套方法论被更优方案替代。例如：五把刀→叙事一致性模型。   │
│ → 走完整双轨 + 外评估（用户反馈/同行评审）。            │
│ → 旧范式存档到"历史范式库"，永久保留。                 │
└─────────────────────────────────────────────────────┘
```

#### 2.2 进化前判决清单

在启动任何 L2 及以上改动之前，必须先过这五道闸门：

```
① 穷尽了吗？
   在现有框架内调整参数（阈值、权重、顺序）能解决问题吗？
   如果能 → 不启动重构。

② 是框架问题还是数据问题？
   失效是因为逻辑不对，还是因为遇到了框架设计时没考虑到的新场景？
   如果是后者 → 先考虑扩展，再考虑重构。

③ 改了之后影响面多大？
   这个组件被哪些其他组件依赖？（查架构清单，第一层）
   改动会连锁影响什么？

④ 有更简单的解决方案吗？
   如果没有独立证据说明新逻辑更好 → 不要动。
   "感觉不够好"不是重构的理由。

⑤ 现在是最佳时机吗？
   技能是否处于稳定使用期？用户是否依赖当前输出格式？
   如果正在被高频使用 → 推迟，或先在沙箱中验证。
```

#### 2.3 不该进化的场景

- ❌ 用户满意度高且没有新的失效信号 → **稳定 > 完美**
- ❌ 仅为了"更优雅"而重构 → **拒绝**
- ❌ 没有明确的对比验证方案 → **拒绝**
- ❌ 上一次重构还在验证期内 → **排队，不叠加重构**

### 第三层：进化执行（Evolution Execution）

#### 3.1 进化沙箱

L3 及以上改动，先在沙箱中验证，不与用户实际交互：

```
沙箱条件：
- 准备50-100份历史PPT诊断记录作为测试集
- 新旧框架各自跑一遍
- 对比维度：问题检出率、误伤率、修正成本、用户满意度
- 沙箱通过 → 进入双轨在线验证
- 沙箱不通过 → 修改候选框架，不进入在线阶段
```

#### 3.2 双轨在线验证

```
双轨验证协议
│
├─ ① 设定验证期
│     最少50次诊断，或最少7天，取长者。
│     覆盖不同场景——不能全是一种场景。
│
├─ ② 每次诊断双轨输出
│     同一条输入，旧框架和新框架各自诊断一遍。
│     记录：哪条轨道发现了问题？修正更精准吗？
│     记录：两条轨道结论不同的情况。
│
├─ ③ 对比维度
│     ├─ 检出率：该发现的问题发现了吗？
│     ├─ 误伤率：不该改的改了吗？
│     ├─ 修正成本：改了几处？改的幅度多大？
│     ├─ 演绎流畅度：改完自己翻一遍，顺吗？
│     └─ 用户接受度：如果用户在交互中给了反馈，记录下来。
│
├─ ④ 分歧裁决
│     两轨结论不同时，不自动选新或旧。
│     回到"如果我是听众"手动判断。
│
└─ ⑤ 渐变切换（仅L3/L4）
      验证期结束后，如果新框架胜出：
      - 第1-3天：新框架处理 30% 的请求（随机抽样）
      - 第4-7天：新框架处理 60%
      - 第8天起：新框架处理 100%
      - 任何一个阶段出现显著恶化 → 暂停、分析、决定
```

#### 3.3 回滚协议

```
触发回滚的条件（满足任一）：
- 双轨验证中新框架的误伤率 > 旧框架的 1.5 倍
- 用户明确对新框架输出表示不满意 ≥3 次（在验证期内）
- 新框架的"演绎流畅度"不通过率 > 旧框架的 2 倍

回滚操作：
- 恢复旧框架为主逻辑
- 新框架存档到"历史演化记录"，标注"验证未通过"
- 记录回滚原因和对比数据
- 版本号不递增（这次重构视为未发生）
```

### 第四层：进化治理（Evolution Governance）

#### 4.1 版本体系

```
v[major].[minor].[patch]

patch: L1 案例更新、措辞调整
minor: L2 框架微调（新增诊断维度、调整流程、调整权重）
major: L3/L4 框架重构、范式替换

版本记录格式：
---
version: 1.1.0
date: 2026-08-15
type: L2 框架微调
change: 五把刀增加第六维度"交互设计"（PPT中的互动环节设计）
reason: 过去30天慢车道日志显示，约12%的诊断涉及PPT互动环节
        缺失的问题，当前五把刀无法归类
sandbox: 50条测试，检出率从88%提升到94%，误伤率持平
dual_track: 7天/58次诊断，新框架检测到额外5个旧框架漏掉的问题
user_feedback: 无负面
---
```

#### 4.2 进化日志（Evolution Log）

每次进化尝试（无论成功还是失败）必须记录：

- 触发原因：什么信号、什么场景触发了进化
- 尝试的方案：怎么改的
- 验证方法：沙箱+双轨，还是仅双轨
- 结果：通过/失败/部分通过
- 关键教训：这次进化教会了我们什么
- 对进化协议的影响：协议本身需要调整吗？

### 第五层：元进化（Meta-Evolution）

> 进化协议本身也是可以进化的。

#### 5.1 元进化触发条件

```
- 进化协议拒绝了一次重构，但后续证明那次重构是必要的
  → 协议的"不该进化"标准需要校准

- 连续两次重构都没通过验证
  → 协议的决策逻辑可能有问题

- 双轨验证通过但上线后用户反馈变差
  → 协议的验证标准不够严格

- 同一个类型的框架问题在不同技能中反复出现
  → 协议缺少某种预防机制

- 技能长期没有进化，不是因为没问题，而是信号未被感知
  → 信号敏感度需要调整
```

#### 5.2 元进化的执行方式

```
① 发现元进化信号 → 记录但不立即行动
② 积累到 ≥3 个同类信号 → 提出元进化方案
③ 在下一次框架进化中，附带给元进化方案开一个"观察窗口"
④ 经过至少两次框架进化周期的对比观察 → 决定是否正式修改元协议
⑤ 元协议修改 → 更新技能模板（影响到后面所有技能的制作）
```

#### 5.3 🔒 不可变核心

有些东西不能被进化掉，否则整个体系崩溃：

```
🔒 不可变核心：

1. 双轨验证原则
   → 新旧并行验证永远不能省

2. 以结果为准原则
   → 不因为"新更优雅"而替换，必须数据说话

3. 可回滚原则
   → 任何改动必须保留回滚能力

4. 历史保留原则
   → 废弃的逻辑不删除，存档备查

5. 人类可干预原则
   → 重大进化必须可以被人类否决

这些不是"规则"，是进化协议能安全运行的底层合约。
```
